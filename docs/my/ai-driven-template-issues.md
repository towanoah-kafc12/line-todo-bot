# AI駆動開発テンプレートの問題点メモ

## 結論

このテンプレートは、要件整理と設計整理の土台としては使えるけど、AI が「次に何をするべきか」を迷わず実行するためのテンプレートとしてはまだ弱い。
理由は、方針や制約は書かれている一方で、実行順、作業単位、停止条件が実装レベルまで落ちていないからだよ。

## 何が足りないか

### 1. `docs/TASKS.md` が抽象的すぎる

今の `TASKS` は人間向けの TODO に近くて、AI がそのまま実装順に変換しにくい。
たとえば「アプリ雛形を追加する」では広すぎて、どのファイルから触るか、何を作れば完了かが分からない。

不足している情報:

- 実行順
- 1 タスクごとの責務
- 変更対象ファイル
- 完了判定
- ブロッカー

### 2. 全体の完了条件はあるが、作業単位の完了条件がない

`docs/DELIVERY.md` には全体の Done Definition はある。
でも、AI が実際に進めるのはもっと小さい単位だから、「このタスクはどこまでやれば終わりか」が必要だよ。

不足している情報:

- config 実装完了の条件
- webhook 実装完了の条件
- テスト追加完了の条件
- docs 更新完了の条件

### 3. 未確定事項が「記録」で止まっていて「行動判断」に繋がっていない

`docs/tracking/common/QUESTIONS.md` に未確定事項を逃がす仕組みはある。
でも、「未決でも進めてよい範囲」と「未決だから止まる範囲」が明文化されていない。

その結果、AI は次のどちらかに寄りやすい。

- 不必要に止まる
- 勝手に強く仮定して進める

### 4. 実装手順と設計書の間を埋める文書がない

`PROJECT` `ARCHITECTURE` `TECH-STACK` はあるけど、そこから実装に落とす橋渡しが弱い。
AI 駆動では、設計書だけだと抽象度が高くて、毎回 AI 側が実装計画を再構成する必要が出る。

足りない文書の例:

- `implementation-plan.md`
- `runbook.md`
- `work-packages.md`

### 5. 入出力の定義が task 単位で弱い

AI は「何を入力として読んで、何を出力として残すか」が明確だと強い。
今のテンプレートは repo 全体のルールはあるけど、各作業の入力と成果物の対応が薄い。

たとえば足りないのはこういう情報だよ。

- 入力: `docs/requirements.md`, `docs/setup.md`
- 出力: `app/src/config/env.ts`, `.env.example`
- 検証: `npm run test -- env`

### 6. 優先順位と依存関係が弱い

AI は parallel に進められる部分と、順番を守るべき部分を区別できると安定する。
でも今のテンプレートは、「何が前提になって次の作業が解放されるか」が弱い。

たとえば本来はこういう依存関係が必要だよ。

- env 設計が決まる
- webhook handler を作る
- command parser を作る
- Todoist adapter を作る
- integration test を追加する

### 7. 変更追跡はあるが、実装サイクルのループが弱い

このテンプレートは変更追跡の思想は良い。
でも AI 駆動では、次のループを短く回せる形が必要だよ。

必要なループ:

1. task を選ぶ
2. 前提 docs を読む
3. 実装する
4. 検証する
5. docs / tracking を更新する
6. 次タスクを明示する

今は 6 が弱いから、各ターンの終わりに AI が次の行動を再解釈する負担が大きい。

## このテンプレートの良い点

問題だけじゃなく、土台として良い点もある。
理由は、改善時に残すべき部分を切り分けるためだよ。

- `AGENTS.md` の常設ルールがある
- `BOOTSTRAP` `PROJECT` `TECH-STACK` `ARCHITECTURE` `TASKS` に役割分担がある
- `QUESTIONS.md` と `BUGS.md` に逃がす思想がある
- docs 更新責務が明記されている

つまり、全部作り直す必要はなくて、足りないのは「実行可能な粒度への分解」だね。

## 本質的な問題

本質は、現在のテンプレートが「設計管理テンプレート」としてはそこそこ良い一方で、「AI 実行オーケストレーションテンプレート」としては不足していることだよ。

設計管理テンプレートが重視するもの:

- 何を作るか
- 何を記録するか
- 何を更新するか

AI 実行テンプレートが追加で必要とするもの:

- 次に何をするか
- どの順でやるか
- どこまでやれば止まるか
- 未確定でも進めてよい範囲はどこか

今は前者に寄っていて、後者が薄い。

## 追加で明確になった問題

### 8. AI が自律的、主体的に次の一手を決める設計になっていない

これはかなり重要な問題だよ。
今のテンプレートだと、AI は与えられた質問には答えられるし、与えられた task も進められる。
でも、repo 全体を見た上で「では次に何をやるべきか」を自律的に決めて、そのまま着手しやすい形にはなっていない。

理由は次のとおりだね。

- 次アクションを決める優先順位ルールがない
- 実装開始条件と着手順が task 単位で固定されていない
- 未確定事項が残るときに、どこまで自走してよいかの境界が弱い
- 途中完了時に「次に選ぶべき task」が明示されない

その結果、AI は毎回こういう再判断を強いられる。

- 今は docs を触るべきか、コードを書くべきか
- 未確定事項があるが進めてよいか
- 複数タスクがある中で、どれが最優先か
- 今の作業が終わったあと、次はどこへ進むか

これは AI の能力不足というより、テンプレート側が判断材料を十分に与えていないのが原因だよ。

言い換えると、今のテンプレートは「指示に従って動く AI」には向いているけど、「repo の状態を見て主体的に次を進める AI」にはまだ最適化されていない。

### 9. 適度なコミット、プッシュ、ブランチ運用が作業フローに組み込まれていない

これも AI 駆動ではかなり大きい問題だよ。
今回の要件には「こまめに Git コミット、プッシュしてほしい」とあるのに、テンプレート側にはその判断基準と実行タイミングがない。

その結果、AI は次の判断を毎回その場で考えることになる。

- どの粒度でコミットするべきか
- どのタイミングでプッシュするべきか
- main で続けてよい段階か、ブランチを切るべき段階か
- docs 変更だけでもコミット対象にするべきか
- 実装途中で未完成でも保存コミットを切るべきか

これは単なる Git 手順不足じゃなくて、AI の作業単位と変更履歴の単位が一致していない問題なんだよ。

不足している情報:

- コミットを切る基準
- プッシュする基準
- ブランチを切る条件
- PR を作る条件
- 途中コミットで許される状態
- コミットメッセージの方針

AI 駆動では、work package の完了とコミット単位が対応している方がかなり安定する。
理由は、実装の区切り、レビュー単位、ロールバック単位が一致するからだね。

理想はこうだよ。

- WP-001 完了で 1 commit
- WP-002 完了で 1 commit
- リモート共有が必要な節目で push
- MVP が一段落したら feature branch + PR に移行

今のテンプレートは docs 更新や task 分解の話はあるけど、Git の保存戦略が設計に入っていない。
だから、AI に「主体的に開発を進めて」と言っても、変更履歴の残し方までは自律しにくい状態になっているね。

### 10. コミットと push を Codex が自然にやり切る運用ルールが弱い

これは 9 と近いけど、もう少し運用面の問題だよ。
テンプレートに Git の概念があっても、「Codex がどの場面で commit し、どの場面で push するか」が行動ルールとして弱いと、結局セッション依存の判断になりやすい。

今回みたいにユーザーが途中で `後でコミットしてね` と補足しないと、次の問題が残るね。

- 変更が長くワークツリーに溜まる
- 端末やセッションが切り替わると継続性が落ちる
- どこまで保存済みかが曖昧になる
- docs 更新とコード更新が別のまま残りやすい

つまり不足しているのは、Git の知識ではなくて、Git を自律実行するための停止条件と保存条件なんだよ。

必要なのは、たとえばこういうルールだね。

- 意味のある作業単位が閉じたら commit する
- リモート共有の価値がある節目で push する
- 一時ログと生成物は commit しない
- commit 前に `git status` で保存対象を確認する

このルールがテンプレートに入っていないと、AI は実装はできても、変更履歴の品質管理まで自然には担いにくいよ。

## 改善方針

### ベスト案

`docs/TASKS.md` を、実行順つきの work package 形式に変える。
理由は、既存ファイルを活かしながら AI の次行動を一意にしやすいからだよ。

持たせたい項目:

- ID
- Phase
- Objective
- Inputs
- Files to touch
- Output
- Validation
- Blockers
- Done condition

### 代替案

新しく `docs/implementation-plan.md` を足す。
理由は、既存の `TASKS.md` を人間向け一覧のまま残したい場合に相性がいいからだよ。

### 補助案

`docs/DELIVERY.md` に作業単位の Done Definition を追加する。
理由は、AI の停止位置を安定させられるからだね。

## 改善後のイメージ

理想は、AI が repo を開いたら次の 1 行をすぐ決められる状態だよ。

例:

- 次の作業は `WP-APP-003 webhook handler 実装`
- 入力は `docs/requirements.md` と `docs/setup.md`
- 変更対象は `app/src/server.ts` と `app/src/line/webhook.ts`
- 完了条件は「署名検証付き POST `/webhook` が通り、テストが 2 本追加されていること」

この粒度まで落ちると、AI はかなり安定して進められる。

## まとめ

このテンプレートの問題は、設計不足ではなく、実行粒度不足だよ。
だから改善の中心は docs の追加量を増やすことではなく、次の行動が決まる形に task と done condition を再設計することになるね。
