あなたは、個人用の小規模アプリを実装するコーディングエージェントです。
目的は、**LINE Bot を UI として使い、Todoist の特定セクションだけを家の共有 TODO として扱える仕組み**を作ることです。

# 目的

IT知識がある利用者（私）と、IT知識がない同居人 A が、家の TODO を共有できるようにしたい。
A は **LINE Bot だけ**で操作できればよい。
私は **Todoist もしくは LINE Bot** で管理できればよい。
データの正本は Todoist とし、**Todoist の特定プロジェクト内の特定セクション**だけを共有対象にする。

# 実現したいこと

A は LINE Bot とのやりとりで、共有 TODO に対して以下をできるようにしたい。

- 一覧を見る
- 追加する
- 完了にする
- 削除する
- 編集する

私は Todoist 本体でも同じ対象を管理できるようにしたい。
つまり、**Todoist 側の対象セクションと LINE Bot 側の表示・操作対象が一致すること**が要件である。

# 基本方針

- LINE は **Reply API 中心**で実装する
- 不要な Push 通知は避ける
- A は LINE だけ使えばよい
- 私は Todoist を通常利用できる
- システム構成はできるだけ小さくする
- 個人利用前提なので、まずは MVP を優先する

# 想定アーキテクチャ

以下のような構成を第一候補とする。

- LINE Messaging API を受ける小さな Web サーバ
- そのサーバが Todoist API を呼ぶ
- 設定として以下を保持する
  - 対象 Todoist project_id
  - 対象 Todoist section_id
  - LINE 利用者の userId と権限

- 永続化が不要なら最小限の設定ファイルで開始してよい
- インフラはいったん考えず、PCでホストしてほしい。
- アプリ側が最低限完成したらAWSでいい感じにホストしたいけど、それはまだ考えなくていい。

# MVP の対象機能

最初の実装対象は次に限定する。

1. `みる`
   - 対象セクションの未完了 TODO 一覧を表示する

2. `追加 {内容}`
   - 対象セクションに TODO を追加する

3. `完了 {番号}`
   - 現在表示中の番号付き一覧の該当 TODO を完了にする

4. `削除 {番号}`
   - 現在表示中の番号付き一覧の該当 TODO を削除する

5. `編集 {番号} {新しい内容}`
   - 該当 TODO の内容を書き換える

# UX 方針

- A は IT に詳しくない前提なので、なるべく単純な操作にする
- 最初は自然言語理解を頑張りすぎず、**明示コマンド型**でよい
- 一覧は必ず番号付きで返す
- `完了 2` のように番号で操作できるようにする
- エラー時は、何がダメかを短く具体的に返す
- 例:
  - 「番号が見つからないよ」
  - 「追加する内容が空だよ」
  - 「この操作は許可されていないよ」

# 権限方針

- A に見せるのは共有対象セクションのみ
- 他の Todoist プロジェクトやセクションは触れさせない
- 必要なら利用者ごとに権限を分ける
- ただし MVP では、私と A の 2人だけを想定してよい

# 実装上の重要方針

## 1. 不明点は都度確認すること

仕様・運用・UI・権限などに不明点が出たら、勝手に強く仮定しすぎず、要所で質問すること。
ただし、**調べれば分かることは先に調べる**こと。
質問は本当に分岐に影響するものだけに絞ること。

## 2. 私の手作業を極力減らすこと

利用者である私の操作が必要な箇所は最小化すること。
自動化できる部分は極力自動化すること。
ただし、外部サービスの仕様上、私が必ず行う必要がある作業は残してよい。
その場合は、**何を・どこで・どの順番で・どう入力するか**を正確に文書化すること。

## 3. 正確なセットアップ手順を必ず調査すること

LINE Developers、LINE Official Account、Todoist API の利用開始に必要な手順は、**必ず公式情報を調査してから**まとめること。
古い情報や推測で済ませないこと。
手順書には以下を含めること。

- 事前に必要なアカウント
- どの画面を開くか
- どの設定を有効にするか
- どの値を控えるか
- どの環境変数に入れるか
- どの URL を設定するか
- 動作確認方法

## 4. 開発前に指示書を作ること

いきなり実装を始めず、まず以下を作成すること。

- 要件整理
- 不明点一覧
- 想定アーキテクチャ
- データフロー
- セキュリティ上の注意点
- セットアップ手順書
- 実装計画
- テスト計画

これらを私に見せてから実装に入ること。

# 技術選定の希望

以下を優先候補とする。

- 言語: TypeScript
- 実行環境: Node.js
- Web フレームワーク: 軽量なものを優先
- データ保存: まずは不要なら持たない。必要なら SQLite
- デプロイ: ローカルで動作確認しやすいことを優先
- 外部公開が必要なら、簡単な候補を比較して提案すること
- 将来的に最小コストでAWSでホストすることを考えたい

ただし、他に明確に有利な選択肢がある場合は、その理由を示したうえで変更提案してよい。

# テスト方針

最低限、以下を確認すること。

- `みる` で一覧取得できる
- `追加` で Todoist の対象セクションに追加される
- `完了` が正しい TODO に作用する
- `削除` が正しい TODO に作用する
- `編集` が正しく反映される
- 想定外入力で分かりやすいエラーになる
- 権限外操作が防げる
- 環境変数不足時に安全に失敗する

可能なら、単体テストと最低限の結合テストも用意すること。

# セキュリティ方針

- トークンやシークレットをコードに直書きしない
- `.env` 利用を前提にする
- `.env.example` にはダミー値だけを書く
- Webhook 署名検証を実装する
- 権限外 userId からの操作を拒否する
- ログに秘密情報を出さない
