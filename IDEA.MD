# AI駆動開発プロジェクト用テンプレート設計ドキュメント

## 1. 結論

このプロジェクトでは、**`AGENTS.md` を唯一の常設エージェント指示ファイルとして中核に置きつつ、将来の skill 管理に耐えられるよう `skills/` ディレクトリを最初から備える**方針を採る。

理由は次の通り。

- 日常運用では `AGENTS.md` だけ見ればよく、運用が散らかりにくい
- ただし将来的には AWS インフラ変更、アプリ機能追加、障害調査などの長い手順を skill として分離したい
- そのため、最初から `skills/` の置き場は作るが、初期段階では skill を最小限に抑える
- `README.md` と `docs/` は人間と AI が共有する事実・設計・作業状況の置き場にする
- `CLAUDE.md` や Cursor 専用 Rules など、ツール固有のファイルは初期テンプレには含めない

この方針により、**シンプルさと将来拡張性を両立する**。

---

## 2. この会話から読み取れる依頼者の目的

依頼者が求めているのは、単なる雛形ではなく、**AWS インフラ開発やアプリ開発に使える AI 駆動開発プロジェクトの開始テンプレート**だと解釈する。

目的はたぶん次の通り。

1. AI コーディングエージェントが最初から迷わず動けること
2. 毎回プロンプトで長々と前提説明しなくてよいこと
3. AWS インフラ開発とアプリ開発の両方に使い回せること
4. いまは軽量に始めたいが、将来的には skill 的な再利用単位も管理できること
5. エージェント向け設定ファイルが増殖しすぎないこと
6. 人間が見ても分かる構造であること

つまり、求めているものは「AI に優しい」だけでは足りず、**人間が保守でき、運用が破綻しにくいテンプレート**である。

---

## 3. テンプレート設計の基本方針

### 3.1 中核方針

- エージェント向けの常設ルールは `AGENTS.md` のみ
- 詳細な設計や仕様や作業状況は `docs/` に置く
- 長い手順や専門知識は将来的に `skills/` へ切り出せるようにする
- ただし初期状態では skill を増やしすぎない
- エージェントが読む順番と参照先を明示する
- 不要な大規模変更を避ける前提を最初から埋め込む

### 3.2 避けたいこと

- `AGENTS.md` にすべてを詰め込むこと
- ツールごとの専用指示ファイルを初期から大量に置くこと
- 一時的なタスクと永続ルールを同じファイルに混ぜること
- ルールだけ立派で運用されないテンプレートになること

## 4. 結論として採用するディレクトリ構造

```text
project-root/
├─ README.md
├─ AGENTS.md
├─ docs/
│  ├─ PROJECT.md
│  ├─ ARCHITECTURE.md
│  └─ TASKS.md
├─ skills/
│  ├─ _shared/
│  │  └─ README.md
│  ├─ aws-infra-change/
│  │  └─ SKILL.md
│  ├─ app-feature/
│  │  └─ SKILL.md
│  └─ incident-investigation/
│     └─ SKILL.md
├─ infra/
├─ app/
└─ scripts/
   ├─ bootstrap.sh
   ├─ lint.sh
   ├─ test.sh
   └─ deploy.sh
```

## 5. この構造の意図

### `README.md`

人間向けの入口。
プロジェクト概要、起動方法、ざっくりした構造説明を書く。

### `AGENTS.md`

エージェント向けの唯一の常設指示ファイル。
作業ルール、読む順番、変更ポリシー、検証ルール、どの skill をいつ参照するかを書く。

### `docs/`

人間と AI が共有する事実の置き場。
仕様、設計、進行状況をここに分けて保持する。

### `skills/`

将来の詳細手順の置き場。
初期段階では空でもいいが、少なくとも構造は用意しておく。
できればサンプルとして 2〜3 個の skill を置き、粒度と命名規則の基準を固定する。

### `infra/`

AWS や IaC の実体を置く場所。

### `app/`

アプリケーション実装を置く場所。

### `scripts/`

セットアップ、テスト、Lint、デプロイなど、エージェントが入口として使えるコマンドをまとめる場所。

- ディレクトリ構造の簡易説明

## Goal

何を作るかを簡潔に書く。

## Scope

-
-

今回やらないこと:

-
-

## Stack

- Language:
- Framework:
- Infra:
- CI/CD:

## Quick Start

```bash
./scripts/bootstrap.sh
./scripts/lint.sh

```

## Directory

## 6.2 `AGENTS.md`

### 役割

エージェント向けの中核ルール。
このテンプレートにおいて最も重要なファイル。

### 書くべき内容

- プロジェクトの目的
- 作業開始時に読む順番
- 変更ポリシー
- テスト・検証ポリシー
- ドキュメント更新ルール
- skill の使い分け
- 禁止事項や注意事項

### 例

```md
# AGENTS.md

## Purpose

このリポジトリは AWS インフラ開発とアプリケーション開発の両方を扱う。
変更では安全性、再現性、レビュー容易性を優先すること。

## Read Order

作業開始時は次の順で確認すること。

1. `README.md`
2. `docs/PROJECT.md`
3. `docs/ARCHITECTURE.md` を必要に応じて確認
4. `docs/TASKS.md` で現在の作業状況を確認

## Working Rules

- 変更前に対象範囲と方針を短く整理する
- 不要な大規模リファクタや無関係な整形はしない
- 既存の命名規則と責務分離を尊重する
- 依存関係の追加は最小限にする
- 破壊的変更は明示する
- 指示がない限り、構造全体を作り直さない

## Validation Rules

- コード変更時は `./scripts/lint.sh` と関連テストを実行する
- インフラ変更時は差分確認を行う
- 失敗時は原因と再現条件を記録する

## Documentation Rules

- 仕様変更時は `docs/PROJECT.md` を更新する
- 構成変更時は `docs/ARCHITECTURE.md` を更新する
- タスク状況が変わったら `docs/TASKS.md` を更新する

## Skills

以下の作業では対応する skill を参照すること。

- AWS インフラ変更: `skills/aws-infra-change/SKILL.md`
- アプリ機能追加: `skills/app-feature/SKILL.md`
- 障害調査: `skills/incident-investigation/SKILL.md`

## Guardrails

- 秘密情報を新規に埋め込まない
- 本番影響がある変更は慎重に扱う
- 説明可能な変更だけを行う
```

## 6.3 `docs/PROJECT.md`

### 役割

プロジェクトの目的、スコープ、完了条件を固定する。

### 書くべき内容

- 目的
- 非目的
- ユーザー
- 成功条件
- 制約
- リスク

### 例

```md
# PROJECT

## Goal

このプロジェクトで実現したいことを書く。

## Non-Goals

今回は対象外とすることを書く。

## Users

誰が利用者かを書く。

## Success Criteria

- [ ] 必須機能が動作する
- [ ] AWS 上でデプロイできる
- [ ] 最低限の検証が通る
- [ ] 運用確認に必要な情報が残っている

## Constraints

- 予算
- 納期
- 既存システム制約
- 利用可能なサービスやランタイム

## Risks

- 認証まわりの複雑化
- 権限過多
- 監視不足
- 環境差異
```

## 6.7 `skills/aws-infra-change/SKILL.md`

### 役割

AWS インフラ変更時の標準手順。

### 書くべき内容

- いつ使うか
- 変更前確認
- 差分確認
- ロールバック観点
- 記録すべきこと

### 例

```md
# AWS Infra Change

## When to Use

- `infra/` 配下を変更する時
- IAM, Network, Compute, Logging, Monitoring に触る時

## Steps

1. 変更対象の環境を確認する
2. 関連ドキュメントを確認する
3. 影響範囲を列挙する
4. 差分確認を行う
5. ロールバック可否を確認する
6. 必要なら `docs/ARCHITECTURE.md` を更新する

## Checklist

- [ ] 本番影響の有無を整理した
- [ ] IAM 権限が過剰でない
- [ ] ネットワーク設定変更の意図が説明できる
- [ ] ログと監視の抜けがない
- [ ] 戻し方がある
```

## 7. `scripts/` に求めること

エージェントが迷わないように、少なくとも以下の入口は揃える。

- `bootstrap.sh`: 初期セットアップ
- `lint.sh`: 静的検査
- `test.sh`: テスト実行
- `deploy.sh`: デプロイ実行またはダミー入口

現時点で中身が未確定でも、最低限 `echo "TODO"` だけの雛形は置いてよい。
理由は、`AGENTS.md` 側から参照できる実行入口を固定したいから。

---

## 8. skill を最初から備えるが、増やしすぎない理由

ここは重要。

このテンプレートは、**最初から複雑にしない**ことが大事。
ただし将来性は必要なので、`skills/` の構造だけは最初から作る。

初期段階では次の 3 skill があれば十分とする。

1. `aws-infra-change`
2. `app-feature`
3. `incident-investigation`

この3つなら、AWS インフラ開発とアプリ開発の主要ユースケースをかなりカバーできる。
逆に、Lambda ごと、VPC ごと、S3 ごと、みたいな細かい skill 乱立は避ける。理由は、責務の分割が細かすぎると管理が破綻しやすいから。

## 9. コーディングエージェントに依頼したい具体タスク

このドキュメントを受け取ったエージェントには、次を実行してほしい。

### 9.1 生成対象

- 上記ディレクトリ構造を作成する
- 各 Markdown ファイルを初期テンプレ内容で作成する
- `scripts/` に最小の実行入口を作る
- `infra/` と `app/` には空ディレクトリまたは `.gitkeep` を置く

### 9.2 守るべき方針

- まず骨組みを作る
- 過剰な実装はしない
- ツール固有ファイルは増やさない
- `AGENTS.md` を最重要ファイルとして扱う
- `skills/` は将来拡張しやすい形にする
- 見た目より保守性を優先する

### 9.3 完了条件

- ディレクトリ構造が揃っている
- `README.md`, `AGENTS.md`, `docs/*`, `skills/*` が最低限の内容で埋まっている
- `scripts/*` が実行入口として存在する
- テンプレートの意図がドキュメントから読み取れる

---

## 10. 実装時の優先順位

1. ディレクトリ作成
2. `README.md` と `AGENTS.md` の作成
3. `docs/` 配下の作成
4. `skills/` 配下の作成
5. `scripts/` の雛形作成
6. 必要なら `.gitkeep` 追加

理由は、まず入口と構造を固定しないと、中身がぶれやすいから。

---

## 11. 最終要約

このテンプレートの本質は、**`AGENTS.md` を中核にした軽量運用を守りながら、`skills/` による将来拡張に最初から耐える二層構造**にある。

要するに次のように設計する。

- `AGENTS.md`: 常設ルール
- `docs/`: 事実、仕様、設計、タスク
- `skills/`: 長い手順の拡張ポイント
- `scripts/`: 実行入口
- `infra/`, `app/`: 実体
